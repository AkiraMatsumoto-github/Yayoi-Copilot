# レシピ機構 — 設計書（ハイブリッド: 固定操作 ＋ AI）

## 1. 目的

決まった道順の操作（メニュー遷移など）は、毎回 DOM 抽出＋Claude問い合わせを行わず、
**あらかじめ定義した手順（レシピ）を決定的に実行**する。判断・読み取り・曖昧な指示が
必要な部分だけ既存の AI ループに任せる。

### 解決したいこと
- **速度**: 固定操作は1手ごとの数秒（ネットワーク往復＋LLM推論）を消す
- **コスト**: 固定操作のトークン消費をゼロにする
- **再現性**: LLM のブレを排し、同じ操作は毎回同じ結果にする

### 壊さないこと
- **頑丈さ**: 弥生がUIを変えても止まらないよう、レシピが失敗したら AI が引き継ぐ（自己修復）

---

## 2. 全体方針

操作は2層に分ける。

| 層 | 担当 | LLM | 例 |
|---|---|---|---|
| ナビゲーション（道順） | レシピ（決定的） | 不要 | メニュー→レポート・帳簿→仕訳帳 |
| 判断・読み取り | AIループ | 必要 | 「どの行か」「内容を5件報告」「曖昧な指示」 |

ひとつの指示はこの2層を**つなげて**処理できる。
例:「仕訳帳を開いて最近5件報告して」
→ 開くところはレシピ（速い・無料）→ 報告だけ AI。

## ⭐ 現在地の特定 — 画面モデル（設計の土台・最重要）

**「今どの画面にいるか」が分からなければ、レシピは始められない。** 同じ「仕訳帳を開いて」でも、
マイポータルにいるのか、すでに青色申告オンラインにいるのか、別の帳簿を開いているのかで
押すべき場所はまったく変わる。だから現在地の特定を**全ての操作の起点（ステップ0）**に置く。

### 3.0 何が問題か
- 弥生は複数プロダクト（マイポータル / やよいの青色申告 オンライン / Misoca …）をまたぐ
  **多ページ型Webアプリ（MPA）**。画面遷移ごとにページが読み込まれ、URLが変わる。画面ごとに
  メニュー構成も違う。
- レシピを「マイポータルから始まる固定手順」で書くと、出発点がズレた瞬間に壊れる。
- 「すでに目的の画面にいる」なら、ナビをスキップしていきなり読み取り/操作に入りたい。

### 3.1 URLが第一キー（MPAなので信頼できる）
「現在地はURLで取れるのでは？」── その通り。弥生は **MPAで画面ごとにURLが変わる**ため、
URLは**最も信頼できる現在地の手がかり**になる。拡張は `chrome.tabs.get(tabId).url` で
**DOMを読まずに即・無料**でURLを取れる（`page.url` でも可）。多くの画面は**URL（パス）だけで一意に確定**できる。

ただしURLだけで足りない場面も少しある:
- **モーダル・確認ダイアログ**が開いてもURLは元のまま（ダイアログ表示中はDOMで見分ける）。
- 一覧の**絞り込み・並べ替え**などが同一画面内のAJAX/部分更新で、URLが変わらないことがある。
- パスが似ていて区別したい画面（パラメータ違い等）。

よって **URLを第一キーにしつつ、必要なときだけDOMの目印（本文テキスト・要素）で補う**。
URLで一意に決まる画面はDOMを読まずに確定し（速い）、URLが曖昧なときだけ本文で絞る。

### 3.2 画面モデル（screens）
画面に名前と「見分けるための目印（signature）」を定義する。判定材料は、すでに抽出している
`page = { url, title, text, elements }`。

```js
const SCREENS = [
  {
    id: "myportal",
    name: "マイポータル",
    detect: { urlIncludes: "myportal", anyText: ["マイポータル", "弥生からのお知らせ"] },
  },
  {
    id: "aoiro-home",
    name: "やよいの青色申告 オンライン（ホーム）",
    detect: { urlIncludes: "aoiro", anyText: ["スマート取引取込", "かんたん取引入力"] },
  },
  {
    id: "journal",
    name: "仕訳帳",
    detect: { allText: ["仕訳帳"], anyText: ["取引", "借方", "貸方"], notText: ["かんたん取引入力"] },
  },

  // ── 割り込み画面も一級市民として定義する（重要）──
  // セッション切れ・ダイアログ等を「画面」として検知できれば、暴れずに正しく止まれる。
  {
    id: "login",
    name: "ログイン（要再ログイン）",
    detect: { anyText: ["ログイン", "メールアドレス"], urlIncludes: "login" },
    interstitial: true,
    onEnter: "stop:ログインが必要です。手動でログインしてから再実行してください。",
  },
  {
    id: "dialog",
    name: "確認ダイアログ",
    detect: { hasElement: { role: "dialog" } },
    interstitial: true,
    // 既定の遷移を持たず、検知したら停止して指示を仰ぐ（暴発防止）
  },
];
```

#### signature の書き方
```js
detect = {
  urlIncludes: "journal",      // URL部分一致
  allText:  ["仕訳帳"],         // 全て本文に含まれること（必須条件）
  anyText:  ["借方", "貸方"],   // いずれかが含まれること
  notText:  ["かんたん取引入力"], // 含まれてはいけない（似た画面の区別に使う）
  hasElement: { role: "dialog" }, // 特定要素の存在（任意）
}
```

### 3.3 現在地判定（detectScreen）— 真偽＋具体度で決める

スコアの足し算ではなく、**条件を満たすか（真偽）＋どれだけ具体的か**で決める。曖昧さを残さない。

```
matches(screen, page):
  urlIncludes 指定あり かつ 不一致 → false
  allText の語が1つでも欠ける    → false
  notText の語が1つでも含まれる   → false
  anyText 指定あり かつ 1つも無い → false
  hasElement 指定あり かつ 無い   → false
  → true（全条件クリア）

detectScreen(page):
  cands = SCREENS のうち matches() が true のもの
  if cands 空            → "unknown"
  if cands が1つ         → それ
  if 複数               → 条件数が最も多い（=具体的な）画面を採用
  if それでも同点       → "unknown"（誤判定より安全）
```

- **割り込み画面（interstitial）を最優先で判定**：通常画面より先に login / dialog をチェックし、
  該当すれば `onEnter` に従って停止・報告する。
- `"unknown"` は固定手順に頼らず **AIループに委ねる**（AIは本文を読んで判断できる）。
  誤って固定操作で暴れるより安全。

### 3.4 ページ読み込みの完了を待ってから判定する

MPAなので、リンク/ボタンを押すと**新しいページが読み込まれる**。読み込み途中で `detectScreen` を
呼ぶと、まだ前のページ or 未完成のDOMを読んで誤判定する。だから**遷移を起こしたら、まず読み込み
完了を待ってから**判定する。

```
waitForLoad(timeout=8000):
  chrome.tabs.onUpdated で status === "complete" を待つ
  （＝新しいページの読み込み完了。SPAのような描画安定ポーリングは不要）

detectAfterNav():
  waitForLoad()
  return detectScreen(page を取り直す)
```

- 通常の画面遷移はこれで足りる（URL変化＋load完了が明確な合図）。
- **同一画面内のAJAX部分更新**（一覧の絞り込み等）はページ読み込みが起きないので、その操作では
  `waitFor`（特定の目印の出現待ち）で完了を待つ。
- `goToScreen` の各ホップ後はこの `detectAfterNav` で現在地を取り直す。

### 3.5 ナビゲーションは「画面グラフ」で表す
道順をレシピごとに書き写すのをやめ、**画面間の遷移（edge）**として一度だけ定義する。
こうすると「現在地から目的画面まで」を**その場で経路探索**でき、出発点に依存しなくなる。

```js
const TRANSITIONS = [
  { from: "myportal",   to: "aoiro-home", steps: [ { click: { text: "青色申告", role: "link" } } ] },
  { from: "aoiro-home", to: "journal",    steps: [ { click: { text: "レポート・帳簿" } },
                                                    { click: { text: "仕訳帳" } } ] },
  { from: "journal",    to: "aoiro-home", steps: [ { click: { text: "ホーム" } } ] },
];
```

**経路探索＋自己修正のループ:**
```
goToScreen(targetId):
  for 最大Nホップ:
     here = detectAfterNav()                    # §3.4 ページ読み込み完了を待って判定
     if here が interstitial（login/dialog）: onEnter に従い停止・報告 → return
     if here == targetId: return 成功
     if here == "unknown": return AIに委ねる
     edge = BFSで here→target の最初の1ホップを選ぶ
     if edge なし: return AIに委ねる
     edge.steps を実行 → 読み込み完了を待つ        # 次ループ冒頭の detectAfterNav で再判定
```

各ホップの後に**必ず再判定**するのがポイント。想定外の画面に飛んでも、次のループで現在地から
経路を引き直すか、無理なら AI に切り替わる。割り込み画面（セッション切れ等）を踏んだら暴れずに止まる。
「固定手順をなぞる」のではなく「現在地を見て一歩進む」を繰り返すので、途中のズレを吸収できる。

### 3.6 これで「レシピ」は薄くなる
ナビが画面グラフに集約されるので、レシピ本体は **「目的画面 ＋ そこでやること」** だけになる。

```js
{
  id: "report-journal",
  triggers: [ /仕訳帳/ ],
  goto: "journal",        // ← 現在地から自動で到達（グラフ経由）
  then: { ai: "original" }, // 到達後にやること（読み取り/操作/AI）
}
```

「道順」は §3.5 のグラフ、「どの要素か」は次の §（セレクタ）、「画面でやること」は §5 のステップ、
という役割分担になる。

## 3. 安定したセレクタ戦略（要素の指定方法）

現在 Claude に渡している番号 `[5]` は**画面スナップショットごとの一時的な番号**であり、
次回は別の番号になる。したがってレシピに「5番を押す」と書いてはいけない。

レシピは画面が多少変わっても残りやすい**安定した目印**で要素を指す。優先順位:

1. 表示テキスト（完全一致 → 部分一致）: 例 `"仕訳帳"`
2. `aria-label` / `role`
3. リンクの `href` パターン
4. （最後の手段）CSSセレクタ

要素を探す処理はページ内で `chrome.scripting` で実行し、見つけた要素の画面座標を返す。
実際のクリックは従来どおり `chrome.debugger` の trusted イベントで行う（合成イベントだと
弥生のJSハンドラが反応しないため、ここは AI ループと同じ方式を共有する）。

## 4. レシピの書き方（記法仕様）

レシピは `extension/recipes.js` に**データ（JSONで表せる構造）**として書く。関数は持たせない
（編集を簡単に・将来UIから編集可能に・安全に保つため）。動的な値は「テンプレート展開」、
本当に動的な判断は「AIステップ」に逃がす。

レシピ1件の全体像:

```js
{
  id: "report-journal",          // 一意なID
  name: "仕訳帳を開いて報告",    // 表示用の名前
  triggers: [ /仕訳帳/ ],        // この指示にマッチしたら採用（後述）
  goto: "journal",               // （任意）到達したい画面。現在地からグラフ経由で移動（§3.5）
  then: { ai: "original" },      // 到達後にやること。単一アクション or ステップ配列（§4.2）
}
```

評価は配列の上から順。**最初にトリガーが一致したレシピ**を採用する。
`goto` を省けばその場で `then` を実行、`then` を省けば `goto` 到達で完了。

### 4.1 ターゲット指定（最重要）— 「どの要素か」

click / input / waitFor などは「どの要素を対象にするか」を **target オブジェクト**で表す。
ここがレシピ記法の心臓部。**一時的な番号は使わず、画面が変わっても残る目印**で指す。

```js
target = {
  // ── 目印（いずれか1つ以上。複数指定はAND） ──
  text:  "仕訳帳",        // 表示テキスト
  match: "contains",      // "exact" | "contains"（既定: contains）
  role:  "button",        // アクセシビリティのロール（button/link/tab/menuitem…）
  name:  "保存",          // role と併用するアクセシブルネーム
  aria:  "閉じる",        // aria-label
  href:  "/journal",      // <a> の href 部分一致
  css:   ".btn-primary",  // 最後の手段（CSSセレクタ）

  // ── 絞り込み・あいまいさ解消 ──
  within: { text: "アスクル", closest: "tr" },  // この範囲内だけを探す（後述）
  nth:    0,              // 何番目か（0始まり）。複数一致を「意図的に」選ぶときだけ明示する
  visibleOnly: true,      // 画面に見えている要素だけ（既定: true）
}
```

**指定の優先順位（壊れにくい順）**: `text` / `role`+`name` / `aria` / `href` ＞ `css`。
できるだけ人間が画面で読む言葉（text・name）で書く。`css` はそれらで取れないときの逃げ道。

#### あいまいなクリックは黙って実行しない（安全規則）
`text:"編集"` のように複数の要素に一致しうる目印は事故のもと。**click は「対象が一意に決まること」を
前提**とし、解決時に**複数ヒットしたら失敗扱い**にする（既定で先頭を選ばない）。

- 複数一致したいときは **`within`（範囲を絞る）か `nth`（位置を明示）で必ず一意化**する。
  `nth` を明示した場合のみ、その位置の要素を意図的に選ぶ。
- 一意化されずに複数ヒット → そのステップは失敗 → `onFail`（既定 "ai"）でAIへ。**間違った要素を
  押すより、止まってAIに渡す方が安全**。
- `waitFor`（存在確認）は複数ヒットでも可（「あるか」を見るだけ）。一意性を要求するのは click /
  input / select。

**`within`（範囲スコープ）が肝**: 一覧・グリッド（仕訳帳など）で「アスクルの行の編集ボタン」を
押すといった操作に必須。
- `within` をまず解決して**範囲となる要素**を1つ決め、その**子孫の中だけ**で本体の target を探す。
- `closest` を付けると、`within` の目印にマッチした要素から**最も近い祖先**（CSSセレクタ、例 `"tr"`）
  まで遡って範囲にする。「"アスクル"というセル → その行(tr)全体」を範囲にできる。

例:「アスクルの行の "編集" を押す」
```js
{ click: { text: "編集", within: { text: "アスクル", closest: "tr" } } }
```
例:「一覧の3行目を押す」（位置指定）
```js
{ click: { css: "table tbody tr", nth: 2 } }   // 0始まりなので3行目=2
```

> 実装メモ: target の解決はページ内で `chrome.scripting` が行い、**見つけた要素の中心座標を返すだけ**。
> 実際のクリックは AI ループと共通の `chrome.debugger`（trusted イベント）で行う。二重実装しない。

### 4.2 ステップ種別

各ステップはオブジェクト1つ。キーで種別が決まる。

| 種別 | 形 | 意味 |
|---|---|---|
| `click` | `{ click: <target> }` | 要素を探してクリック |
| `input` | `{ input: { target?, label?, value } }` | 入力欄に文字入力。`label` 指定でラベル名からフィールドを特定可 |
| `select` | `{ select: { target, option } }` | プルダウンで選択 |
| `waitFor` | `{ waitFor: { target?, gone?, url?, timeout? } }` | 目印が**出現**/`gone`で**消失**/`url`で遷移を待つ。`timeout` ミリ秒（既定8000） |
| `expect` | `{ expect: <target> }` | 目印が無ければ**失敗**にする。画面確認・安全確認に使う |
| `scrollTo` | `{ scrollTo: <target> }` | 要素まで스クロール |
| `ai` | `{ ai: "original" }` または `{ ai: "サブ指示" }` | 残りを AI ループに委譲 |

#### 共通オプション（どのステップにも付けられる）

```js
{ click: {...},
  optional: true,        // 見つからなくてもスキップ（失敗扱いにしない）
  onFail: "ai",          // 失敗時の挙動: "ai"(既定,AIに引き継ぐ) | "abort"(中止) | "skip"
  mutates: true,         // データを変更する操作（書き込みガードの対象。後述）
  note: "○○のため",      // 任意のメモ
}
```

`ai: "original"` =「ここから先は**元のユーザー指示**で AI ループを回す」。レシピで正しい画面まで
来ているので、AI は「もう開いている」と認識して読み取り・報告だけ行う。
文字列で別の指示（例 `"表示中の仕訳を5件、表で報告して"`）に差し替えることもできる。

### 4.3 パラメータとテンプレート展開 — 指示から値を取り込む

トリガーの正規表現に**名前付きグループ**を書くと、その値が `params` になる。
ステップ内の文字列は `{{名前}}` で展開される。

```js
{
  id: "filter-journal-by-month",
  name: "仕訳帳を月で絞り込む",
  triggers: [ /仕訳帳.*(?<month>\d{1,2})月/ ],   // 「仕訳帳を5月で…」→ params.month = "5"
  steps: [
    { click: { text: "レポート・帳簿" } },
    { click: { text: "仕訳帳" } },
    { waitFor: { text: "期間" } },
    { select: { target: { label: "月" }, option: "{{month}}月" } },   // ← 展開される
    { ai: "original" },
  ],
}
```

- 展開できるのは `text` / `name` / `aria` / `value` / `option` / `href` などの文字列フィールド。
- 値が取れなかった `{{名前}}` が残る場合はそのステップを失敗とし、`onFail`（既定AI）に従う。

---

### 4.4 トリガー（指示 → レシピの判定）

```js
triggers: [
  /仕訳帳/,                              // 単純キーワード
  /(?<entity>.+?)の(?<field>摘要|金額)を(?<value>.+?)に(直し|変更|修正)/,  // 名前付きで値取得
]
```

- 配列のうち**どれか1つでも**マッチすればそのレシピを採用。
- マッチした正規表現の名前付きグループが `params` になる。
- 表現が大きくぶれてどのレシピにも一致しないときは、**そのまま AI ループに流れる**ので破綻しない。

### 4.5 書き込み安全（mutates）

データを変える操作には必ず `mutates: true` を付ける。原則は **二段構え**:

**① 実行前に確認ゲート（最重要）**
`mutates: true` のステップは、実行する前に**何をするか・どの値を入れるかをサイドパネルに提示し、
ユーザーのOKを待つ**。値が `{{params}}` 由来のときは特に重要 ── トリガーの正規表現は
値を取り違えうる（例 `(?<value>.+?)に` の境界ミス）ので、**入れる値そのものを人間に見せて確認**する。

```
摘要を「6月分 消耗品」に変更します。よろしいですか？   [実行] [中止]
```
- OKされたら実行、中止なら停止。既存の「中断」UI（sidepanel）と同じ仕組みに乗せる。
- 確認は値が確定した直前に出す（複数の mutates があればそれぞれ、またはまとめて1回）。

**② 最後の「保存」は人間**
- レシピは**最後の「保存」を含めない**。入力・選択まで行い、保存ボタンは押さず、
  「内容を確認して保存してください」と報告して止まる（最終確定は人間）。
- どうしても保存まで自動化したい場合のみ、ユーザー指示に明示の保存語（「保存して」等）がある
  トリガーに限定し、`mutates: true` の保存ステップを置く（この場合も①の確認ゲートを通す）。
- 読み取り系レシピには `mutates` を一切付けない（付け忘れ＝安全側）。

### 4.6 実例集

> レシピは **`goto`（目的画面）＋ `then`（到達後にやること）** で書く。道順は §3.5 の画面グラフが
> 現在地から自動で引くので、レシピに道順を書かない。`then` は単一アクションか、ステップ配列。

#### ① ナビ＋読み取り（最頻パターン）
「仕訳帳を開いて最近5件報告して」
```js
{
  id: "report-journal",
  name: "仕訳帳を開いて報告",
  triggers: [ /仕訳帳/ ],
  goto: "journal",            // 現在地がどこでもグラフ経由で仕訳帳へ
  then: { ai: "original" },   // 「最近5件報告」はAIが画面を読んで実施
}
```

#### ② 純ナビのみ（AI不要・最速）
「やよいの青色申告オンラインを開いて」
```js
{
  id: "open-aoiro",
  name: "やよいの青色申告 オンラインを開く",
  triggers: [ /(青色申告|やよいの青色).*(開|起動|見せ)/ ],
  goto: "aoiro-home",         // 到達したら完了（then 無し）
}
```

#### ③ パラメータ＋到達後の操作
「仕訳帳を5月で絞り込んで」
```js
{
  id: "filter-journal-month",
  name: "仕訳帳を月で絞り込む",
  triggers: [ /仕訳帳.*(?<month>\d{1,2})月/ ],
  goto: "journal",
  then: [                     // 到達後の操作はステップ配列で
    { waitFor: { text: "期間" } },
    { select: { target: { label: "月" }, option: "{{month}}月" } },
    { click: { text: "表示", role: "button" } },
    { ai: "original" },
  ],
}
```

#### ④ グリッドの行を編集（書き込み・保存はしない）
「アスクルの摘要を『6月分消耗品』に直して」
```js
{
  id: "edit-row-memo",
  name: "仕訳の摘要を修正",
  triggers: [ /(?<entity>.+?)の摘要を(?<value>.+?)に(直し|変更|修正)/ ],
  goto: "journal",            // まず確実に仕訳帳へ（現在地不問）
  then: [
    // 「アスクル」を含む行の中の「編集」を押す（within+closestで行スコープ）
    { click: { text: "編集", within: { text: "{{entity}}", closest: "tr" } }, mutates: true },
    { waitFor: { text: "摘要" } },
    { input: { label: "摘要", value: "{{value}}" }, mutates: true },
    // ★保存はしない。人間に最終確認させる
    { ai: "入力内容（摘要を {{value}} に変更）を画面で確認できる状態にして、保存は行わず『内容を確認して保存してください』と報告して" },
  ],
}
```

> ④は `goto` で現在地を問わず仕訳帳に揃えてから操作する。`mutates` を付けた操作のみデータに触れ、
> **保存ボタンは押さない**（最終確定は人間）。行が見つからない・画面が違うといった場合は各ステップの
> `onFail`（既定 "ai"）でAIが引き継ぐ。

## 5. 制御フロー

現在地の特定（ステップ0）を起点に、目的画面まで自己修正で移動し、到達後の操作を行う。

```
runAgent(task):
  recipe = matchRecipe(task)            # triggers を順に評価、最初の一致
  if recipe なし:
      → runAiLoop(task)                 # 従来どおり全部AI
      return

  # ── ステップ0: 現在地を見て目的画面へ ──
  if recipe.goto:
      ok = goToScreen(recipe.goto)      # §3.5 detectAfterNav→経路探索→1ホップ→再判定…の繰り返し
      if not ok:
          → runAiLoop(task)             # 現在地不明 or 経路なし → AIに委ねる
          return

  # ── 到達後の操作（then）──
  for step in (recipe.then を配列化):
      if step.ai:
          → runAiLoop(step.ai === "original" ? task : step.ai)
          return
      try:
          execStep(step)                # click / input / select / waitFor / expect（決定的）
      except 見つからない / タイムアウト:
          step.onFail に従う（既定 "ai"）→ runAiLoop(task) で自己修復
          return

  done("完了しました。")                # then が純粋な操作のみ（AI無し）なら完了報告
```

ポイント:
- **ステップ0で必ず現在地を特定**してから動く。出発点がどこでも目的画面に揃う。
- 各ホップ後に再判定するので、途中で想定外画面に飛んでも経路を引き直すか AI に切り替わる。
- レシピのどこかで目印が見つからなければ **AI ループにフォールバック**（壊れても止まらない）。
- ページ読み込み完了（`waitForLoad`）/ `waitFor` で遷移を待つため、固定 `SETTLE_MS`（1.5秒）より速くかつ確実。

## 6. ルーティング（指示 → レシピ判定）の方針

**まずはコード内のローカル一致（triggers）で判定する。** LLM を使わないので速く・無料。

- 採用理由: 「仕訳帳」「仕訳帳を開いて」「仕訳帳見せて」程度のゆらぎは正規表現で十分吸収できる。
- 限界と対策: 表現が大きくぶれてどのレシピにも一致しない場合は、そのまま **AI ループに流れる**ので
  破綻しない（最悪でも従来どおり動く）。
- 将来の拡張: ゆらぎが問題になったら「1回だけ分類用に Claude を呼ぶ」Tier を足す
  （毎ステップではなく指示ごとに1回なのでコストは小さい）。今は入れない（YAGNI）。

## 8. 段階導入

1. **Phase 0（土台）**: **現在地特定**を先に作る。`SCREENS` 2〜3画面（マイポータル / 青色申告ホーム /
   仕訳帳）＋割り込み画面（login / dialog）＋ `detectScreen`（URL第一＋具体度）＋ `detectAfterNav`
   （ページ読み込み完了待ち）。検証: 各画面でサイドパネルに現在地名が正しく出るか。遷移直後でも
   読み込み完了を待って正しく判定するか。セッション切れを login として検知できるか。
   ※ ここが全ての前提なので最初に固める。
2. **Phase 1**: 画面グラフ＋`goToScreen` ＋レシピ1本（「仕訳帳を開く」）。
   - `resolveTarget` / `waitFor` / `execStep` / `runRecipe` / `runAiLoop` 切り出し。
   - 検証: **どの画面から始めても** AI を呼ばずに仕訳帳へ到達するか。失敗時に AI フォールバックするか。
3. **Phase 2**: 画面・遷移・レシピを追加（契約管理、各帳簿、設定画面など）。
4. **Phase 3**: `input` の書き込み系レシピ（最後の保存は人間 or 明示確認）。

## 9. 留意点

- **現在地判定の精度がシステム全体の信頼性を決める。** signature は似た画面を取り違えないよう、
  `allText`（必須語）と `notText`（除外語）で十分に区別する。`unknown` は誤動作より安全（AIに回る）。
- 画面定義・遷移は弥生のUI変更に追従が必要。ただし `detectScreen` が `unknown` を返せば AI に切り替わる
  ため、定義漏れがあってもユーザー操作は止まらない。
- レシピは「目印」で要素を指すため、ラベル文言が変わると個別に直す必要がある。これも失敗時は AI が拾う。
- `click` はあくまで `chrome.debugger`（trusted）で行う。`resolveTarget` は座標を返すだけにして、
  AI ループとクリック実装を共有する（二重実装を避ける）。
- 書き込み系レシピは、AI ループと同じく「明示指示がなければ実行しない」原則を守る。

## 10. 今後の改善（検討事項）

本体仕様には入れないが、実装が進んだら取り込みたい改善点。優先度順ではなく、必要になった時点で着手する。

### 10.1 グリッド行の現実（Phase 3 で要・実地調査）
④のグリッド編集は `closest: "tr"` を前提にしているが、弥生のグリッドが本物の `<table>` とは限らない
（`role="row"` の div などの可能性）。また**仮想スクロール**だと対象行が DOM に存在しないことがある。
- 行スコープを画面ごとに設定可能にする（`rowSelector` を screens 側に持たせる、`closest` に
  `role="row"` も許す等）。
- `resolveTarget` に**スクロールしながら目的行を探す**能力を足す。
- → Phase 3 着手前に、実際の仕訳帳グリッドの DOM 構造を調査してから設計を確定する。

### 10.2 ナビをもっと頑丈・省力に
- **URL直行 edge**: メニューを辿るより `location.href = 既知URL` で一発移動できる画面がある。
  `goToScreen` の手段に「URL遷移」を持たせると経路が短く堅くなる。MPAなので `location.href` で
  普通にそのページが読み込まれ、内部状態の心配も小さい（むしろ最も確実な移動手段になりうる）。
- **`from: "*"` 遷移**: 左メニューは多くの画面で共通。全ペアの edge を書くと組合せ爆発するので
  「メニューが見える画面ならどこからでも」を1本で表せる仕組みを入れる。

### 10.3 レシピ作成を支える仕組み
- **ドライラン（プレビュー）モード**: 実際に押さず「ここを押す」とハイライトだけする。レシピを
  安全に作り込め、`resolveTarget` の当たり外れも目視確認できる。authoring の生産性に効く。
- **トリガー優先順位の規約**: 先頭一致なので、汎用 `/仕訳帳/` が具体 `/仕訳帳.*月/` を食う。
  **具体的なトリガーを配列の前に置く**規約＋衝突を警告する簡易リンタを用意する。
